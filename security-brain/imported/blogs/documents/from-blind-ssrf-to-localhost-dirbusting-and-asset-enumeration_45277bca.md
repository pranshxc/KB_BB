# From blind SSRF to localhost dirbusting and asset enumeration

Story about how inconsistent printing of img alt tag in WeasyPrint pdf generator can turn blind SSRF into localhost dirbuster.

HTML to PDF generators are, in general, known to “very challenging” when it comes to security.

If user is able to inject html and/or JS into the html that will be used for generating PDF by the server, SSRF and Local file read vulnerabilities may occur.
Here is a good list of things that could go wrong:

https://book.hacktricks.xyz/pentesting-web/xss-cross-site-scripting/server-side-xss-dynamic-pdf

However, allowing users to inject <img> tags is considered to be “safer” because, even if the PDF generator is initiating server side HTTP requests from privileged environment, the actual SSRF is just the blind one. Well, that’s not entirely true, because you can try to include internal images, like explained in this example, and if the server is vulnerable you’ll be able to extract data you shouldn’t have access to:
https://mike-n1.github.io/SSRF_P4toP2

While working on the private bug bounty program I came across this exact problem.
PDF generator being used was WeasyPrint.
HTML sanitiser only allowed me to inject <img> tags, and I was able to leak some internal images, like apache icons, by injecting a img that looks like this:
<img src=http://localhost/path/to/image>

But I didn’t want to stop there, so I’ve kept exploring different payloads and tags.
And right there I’ve noticed that there is a strange behaviour in WeasyPrint and handling img alt tag.

Compare these two images, that represent two different PDF generated by WeasyPrint. On the left is injected html and on the right is the generated pdf.

Press enter or click to view image in full size
Press enter or click to view image in full size

Can you see the bug?

If image is broken or inaccessible the string you’ve entered as the value of “alt” tag should be printed instead of an image. So nothing can go wrong there?
Well, not with WeasyPrint!
With WeasyPrint, if img src= endpoint is accessible by server (doesn’t return 404) alt tag is not printed. Is it a valid image or not? That doesn’t matter and that’s the problem — the core of this bug.
If we try to do something like this:
<img src=http://localhost/.git/HEAD alt=Blind?>
and send it to PDF generator.
If the server returns “Blind?” in our PDF that means that .git/HEAD is not accessible form localhost.
However, if we receive and empty PDF, that means that .git/HEAD exists and it can be accessed from the environment where WeasyPrint was running.

From this point. It’s pretty straightforward how bad this can be.
We can now run enumeration and dirbusting of the internal systems. And to make things even worse, we can do this free of charge and let the server enumerate itself and send us the results in the generated pdf.
Just send the following html and observe the generated PDF:
<img src=”http://localhost/secure_login” alt=”secure_login”>
<img src=”http://localhost/secureauth” alt=”secureauth”>
<img src=”http://localhost/secured” alt=”secured”>
<img src=”http://localhost/secureform” alt=”secureform”>
<img src=”http://localhost/secureprocess” alt=”secureprocess”>
….
You get the idea. If you don’t see secureauth, secureprocess or secured in your PDF, you’ll know that those endpoints exist.

Per cost of one request, you’ve just tricked the server into running hundreds or thousands of requests giving you back all the juicy information about internal infrastructure.
