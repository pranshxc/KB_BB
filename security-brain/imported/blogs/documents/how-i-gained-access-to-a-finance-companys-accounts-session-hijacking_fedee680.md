In a web application security test, I gained access to hundreds of accounts that belonged to a finance company. The vulnerability occurred due to insecure session token generation.

Hey everyone. I’m Talha Karakumru. I’ve been doing web application security research and penetration testing for 3 years. I’ve tested hundreds of web applications and APIs during that time. I love digging into authentication and authorization flaws in applications.

In a web application security test, I found a session hijacking vulnerability that occurred due to insecure token generation.

Let’s Talk about the Vulnerability

In the login process, the API was assigning each user a session token that had a low entropy value. “Low entropy” means that the value can somehow be predicted by analyzing thousands of values that are generated by the vulnerable function.

At first, I didn’t notice that the session tokens were not actually random. When I looked at the tokens after generating a few of them, I thought that the tokens were just hash values of some data. I skipped the analysis process at that moment.

After a while, I thought I should’ve focused on the token generation process. At first, I used Burp Suite’s repeater to resend the login request multiple times manually. I realized that some parts of the tokens stayed the same; the other parts were either just incremental or had a few different values.

For the privacy of the company, I anonymized the write-up.

First of all, let’s split a token into 4 parts.

#1: 51000

#2: 1L3IR

#3: 1V0B

#4: 01

#1 was either 51000 or 52000.

#2 was incremental and didn’t change so often. There was no need to brute force it.

#3 was incremental and changed every time, but the length of the value wasn’t enough to prevent brute force attacks.

#4 was either 00 or 01.

In this case, you might have thought that brute forcing part #3 should’ve took too much because there’s 10 digits and English alphabet contains 26 characters.

So, each character in part #3 could have 36 different values. Considering that part #3 had 4 characters, there was 36 x 36 x 36 x 36 = 36⁴ = 1679616 permutations.

The question was “did we really have to brute force from 0000 to ZZZZ?”.

Let’s assume that we got a token like this: 52000 1L3IQ ZLKI 00.

We could guess that 52000 1L3IQ ZLKH 00 has already been taken by another user because ZLKH comes before ZLKI. We could know that hundreds and even millions of users have already logged in, but we could not know how many users have logged in after we’ve done.

Validating Session Tokens

When we sent a valid context (session token) without specifying any service, the API returned UNDEFINED_SERVICE as a response. But when we sent an invalid context, the API returned INVALID_SESSION as a response.
